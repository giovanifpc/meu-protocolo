# Meu Protocolo — Contexto do Projeto

## O que é este app

SaaS B2B2C de gestão para **personal trainers autônomos brasileiros**. O profissional assina um plano, gerencia seus alunos e entrega um app com branding próprio (white-label lite) ao aluno final. Diferencial central: suporte 24/7 via IA, sem fila de ticket.

Produto **totalmente separado da marca Fox Performance** — projetos distintos, sem mistura de dados, marca ou branding.

- **Documento master (planejamento de negócio):** `C:\Users\Giovani\Documents\Meu Ciclo\MEU-PROTOCOLO-MASTER.md` — ler antes de qualquer sessão de desenvolvimento. Decisões estratégicas ali são definições fechadas, não sugestões.
- **Roadmap pós-desenvolvimento (`roadmap-pos-dev.md`):** plano de validação com usuários reais (Cliente 0 → Programa Founder → padrões em 3+ relatos → só então evoluir), **fundido em 2026-07-23** com um segundo roadmap de crescimento mês a mês (metas de assinantes/receita, internacionalização, visão de 3 anos) — ver Fases 11/12 e a seção final "Visão de longo prazo" do documento. Na prática, as Fases 2/3 (Cliente 0) já estão em andamento (ver "Status atual"). Não confundir "documento salvo" com "já estamos no modo congelamento" — o dev normal segue até o usuário sinalizar que o MVP fechou, e nenhuma meta de calendário do roadmap de crescimento justifica pular uma fase de validação.
- **Repositório:** https://github.com/giovanifpc/meu-protocolo
- **Backend:** Supabase (`https://yumqmramxbahkfxsthtt.supabase.co`)
- **Stack:** HTML + CSS + JS puro (sem framework), Supabase Auth + DB (RLS multi-tenant), Mercado Pago (webhook), Claude API (suporte IA), GitHub Pages

## Modelo de dados — multi-tenant

- `professionals` — conta do personal trainer (tenant). Campos de branding: `color_preset` (`azul`/`grafite`/`vinho`/`roxo`/`escuro` — cor livre foi descontinuada em 2026-07-21, ver "Personalização de marca redesenhada" em Status atual), `display_name`. Logo é upload de arquivo (bucket `professional-logos`), não mais URL colada — `logo_url`/`primary_color` (colunas antigas) ficaram sem uso, não removidas. Estado da assinatura: `trial` → `ativo` → `inativo` → `deletado`.
- `students` — aluno vinculado a um `professional_id`. Um aluno pertence a exatamente um profissional.
- Toda tabela de dados de aluno (protocolos, histórico de treino, etc.) referencia `professional_id` **e** `student_id` — RLS filtra por ambos.
- Vocabulário de tabelas em inglês (mesmo padrão adotado no projeto irmão Fox Performance), textos/comentários em português.

## Arquitetura de replicabilidade (item 17 do master doc)

O código-base, sistema de pagamento, chatbot IA e onboarding devem ser projetados desde o início como template replicável por nicho (próxima instância planejada: fisioterapia). Isso significa: nomes de tabela/schema não devem ser específicos de "personal trainer" quando um nome genérico servir igualmente bem, e configurações por instância (MD de contexto da IA, biblioteca de exercícios, branding) devem ficar isoladas, não hardcoded.

---

## Segurança (desde o MVP, não depois)

- Rate limiting em login
- Tokens de sessão com expiração
- Validação de webhook do Mercado Pago via assinatura HMAC
- Headers HTTP de segurança (HSTS, CSP, X-Frame-Options)
- Sanitização de inputs contra SQL/XSS
- Backup automático do banco Supabase

### 2FA obrigatório + códigos de recuperação no painel master (2026-07-23)

Fecha o item 17 do roadmap — o último item de dev pendente da lista original antes de considerar o MVP fechado (ver checklist "Roadmap priorizado" no fim deste arquivo, todos os outros já `~~riscados~~`). Motivação: hoje "segurança do painel master" é literalmente só um e-mail (`meuprotocolo1@gmail.com`) — quem completa o OTP desse e-mail tem CRUD completo de todos os tenants (plano, preço, status, isenção de cobrança, log de conversas de suporte de todo mundo). Um fator só, sem MFA, protegendo dado de todos os profissionais da plataforma.

- **Enforço de verdade é no servidor, não só no client**: as 8 RPCs `master_*` (`supabase_37_master_mfa.sql`) agora exigem, além do e-mail já checado, que o JWT do chamador tenha a claim `aal = 'aal2'` — o Supabase Auth inclui essa claim automaticamente depois que a sessão completa um desafio de MFA (`auth.mfa.verify()`). Checar isso só em `master.html` seria só UX — quem tivesse um access token de sessão aal1 (só o OTP, sem MFA) ainda conseguiria chamar as RPCs direto via REST/curl. Com a claim checada dentro da própria função `SECURITY DEFINER`, isso deixa de ser possível — mesmo padrão de "garantia, não esperança" já usado em outras partes do projeto (veto de técnica por nível, barreira de downgrade de plano).
- **`master.html`**: tela cheia (`#mfaGate`) intercepta o boot antes de carregar qualquer dado. Três estados: (1) nenhum autenticador cadastrado ainda → obrigatório configurar TOTP na hora (QR code + fallback de texto pra digitar manualmente, via `auth.mfa.enroll`/`challenge`/`verify` nativos do Supabase — nenhuma biblioteca de terceiros); (2) autenticador já cadastrado mas sessão ainda em aal1 (login novo) → tela de desafio, pede o código de 6 dígitos; (3) sessão já aal2 → segue direto pro painel. Depois do primeiro cadastro bem-sucedido, gera e mostra 8 códigos de recuperação de uso único automaticamente (só aparecem essa vez).
- **Plano de contingência (autenticador perdido)**: o MFA nativo do Supabase não tem "backup codes" embutido — sem isso, perder o celular travaria o próprio Giovani pra sempre fora do painel master (conta única, sem time de suporte por trás pra resetar). `master_recovery_codes` guarda o hash (sha256, nunca o texto puro) de 8 códigos de uso único, gerados via `master_generate_recovery_codes()` (essa função **exige aal2** — só quem já provou ter o autenticador em mãos pode gerar/ver novos códigos pra si mesmo). Tela de desafio tem um link "Perdi o acesso ao autenticador" que pede um desses códigos em vez do TOTP — validado por `master_verify_recovery_code()` (essa **não** exige aal2, de propósito: é exatamente o caminho de quando o TOTP já era). Código válido aciona a Edge Function `master-mfa-recovery`, que usa a Admin API do Supabase (`service role`, só existe dentro da function) pra apagar o fator TOTP antigo — não existe forma client-side de apagar um fator MFA, só Admin API resolve isso. Depois de removido, `master.html` reabre o cadastro do zero. Botão "Novos códigos de recuperação" no header permite rotacionar o lote a qualquer momento (invalida os anteriores), não só na primeira configuração.
- **`master_recovery_codes`**: RLS habilitada, **zero policy** — mesmo padrão de blindagem total já usado em `messages`/`support_messages` (acesso 100% via RPC `SECURITY DEFINER`, bloqueado por PostgREST direto).
- **Tabela `is_master_email(check_email)`** (helper de `supabase_16`) segue sem uso em lugar nenhum — não fazia parte do escopo desta mudança, não foi tocada.
- **Correção de registro (2026-07-23): `master.html` já vem sendo usado de verdade** — a nota antiga espalhada pelo arquivo ("master.html nunca foi acessado pelo usuário") ficou desatualizada; o usuário confirmou que já loga no painel master normalmente, já cadastrou a Cliente 0 por lá (não pagante, em trial) e já testou a aba "Receita"/faturamento. Ver correção também na seção do chatbot de suporte, mais abaixo. **O que realmente segue sem teste real é só esta implementação nova de MFA** — nem a sintaxe SQL nem o fluxo de TOTP/recuperação foram exercitados contra o banco/Auth real ainda, porque dependem do deploy abaixo (que precisa rodar localmente, mesma limitação de rede de sempre). Validado até aqui só por revisão de código e confirmação via busca (WebSearch/WebFetch na documentação oficial do Supabase) das assinaturas exatas de `auth.mfa.enroll/challenge/verify/listFactors/getAuthenticatorAssuranceLevel` e `auth.admin.mfa.deleteFactor`/`auth.admin.getUserById` usadas. **Pendências de deploy antes do primeiro teste real**: aplicar `supabase_37_master_mfa.sql` (SQL Editor ou `supabase db query --linked --file`, rodando localmente) e `supabase functions deploy master-mfa-recovery` (usa o secret `SUPABASE_SERVICE_ROLE_KEY`, que já existe). **Importante pro usuário**: assim que o SQL rodar, o próximo login em `master.html` vai exigir configurar um autenticador (Google Authenticator/Authy/1Password) antes de ver qualquer dado — tenha o celular em mãos, e salve os 8 códigos de recuperação que aparecem logo em seguida (não aparecem de novo depois).

### Programa de indicação implementado (2026-07-21)

Fecha o item 16 do roadmap (Fase D). Antes de codar, 3 decisões fechadas com o usuário: (1) só quem indica é recompensado (não bilateral); (2) a indicação só vale depois da 1ª cobrança REAL do indicado, não no cadastro (evita gente criando conta só pra fingir indicação); (3) recompensa aplicada via **estorno automático** no Mercado Pago, não manual pelo master — mecanismo explicado e confirmado explicitamente antes de implementar, dado que mexe com dinheiro real.

- **Por que estorno em vez de "pular a próxima cobrança"**: a API de Preapproval (assinatura) do Mercado Pago não tem um endpoint pra simplesmente pular um ciclo. A alternativa óbvia (zerar `transaction_amount` por um ciclo e voltar ao normal depois) exigiria um mecanismo lembrando de reverter na data certa (cron extra, mais um jeito de esquecer/errar timing). Estornar a cobrança que **acabou de ser aprovada** é reativo — dispara direto de dentro do próprio evento de webhook do pagamento, sem depender de prever nada.
- **Schema** (`supabase_33_referrals.sql`): `professionals.referral_code` (único, gerado automaticamente por default em todo profissional, backfill pros que já existiam), `referred_by` (quem indicou), `referral_credit_months` (contador de meses grátis pendentes). Tabela `referrals` (log: referrer_id, referred_id único, `rewarded_at`) — RLS só deixa o **indicador** ver as próprias indicações, o indicado não sabe/não precisa saber. Trigger `create_referral_row()` cria a linha de log automaticamente sempre que `referred_by` é setado (garantia, não depende do client lembrar).
- **`claim_referral(p_code)` (RPC, SECURITY DEFINER)**: o único jeito de vincular uma indicação. Importante: o **código nunca é resolvido pra um id no client** — o client só manda o código, a função resolve o id do indicador internamente, bloqueia auto-indicação (indicar a si mesmo) e só seta `referred_by` se ainda estiver nulo (não dá pra reatribuir depois, nem por alguém mandando update direto via console — a única policy de update em `professionals` não restringe coluna, então sem essa trava na função a coluna seria adulterável).
- **`onboarding.html`**: lê `?ref=CODIGO` da URL, guarda em `sessionStorage` (sobrevive a reload entre os 2 passos do cadastro), e chama `claim_referral()` logo depois de criar a própria linha em `professionals` — non-blocking (se falhar, loga no console mas não impede o cadastro; rastreio de indicação nunca deve travar alguém criando conta).
- **`mercadopago-webhook` (alterada)**: no evento de pagamento aprovado, duas checagens novas: (1) se é a 1ª cobrança aprovada desse profissional (`ultima_cobranca_em` estava nulo antes desse evento) **e** ele foi indicado por alguém com `rewarded_at` ainda nulo → credita 1 mês grátis pro indicador (`increment_referral_credit`); (2) se o profissional que acabou de pagar tem `referral_credit_months > 0` → chama `POST /v1/payments/{id}/refunds` estornando essa cobrança específica e decrementa o crédito (`decrement_referral_credit`). As duas funções de incremento/decremento são RPCs atômicas (update direto, sem ler-then-escrever) pra evitar race condition.
- **`perfil.html`**: card novo "Indique um amigo" — mostra o link (`https://meuprotocolo.app/onboarding.html?ref=CODIGO`) com botão copiar, e quantos meses grátis estão pendentes.
- **2 Edge Functions com deploy pendente**: `mercadopago-webhook` (alterada) e `mercadopago-update-preapproval` (nova, da sessão anterior — upgrade de plano). Nenhuma das duas reflete em produção até rodar `supabase functions deploy` localmente (limitação de rede documentada mais abaixo) — até lá, indicações são registradas normalmente (isso não depende de function nenhuma, só de RPC/trigger no banco), mas a recompensa (crédito + estorno) não vai disparar de verdade.
- **Não testado** — nem localmente nem via curl (mesma limitação de rede). Validado só o que dava pra checar sem deploy: sintaxe TypeScript das 2 functions (via `tsc --noEmit`, sem os erros esperados de módulo/global do Deno) e sintaxe JS das páginas alteradas. **Antes de confiar nisso em produção**: aplicar o SQL (`supabase_33_referrals.sql`) via SQL Editor ou `supabase db push` local, fazer os 2 deploys de function, e testar o fluxo completo com 2 contas de sandbox (indicar, cadastrar, cobrar o indicado, confirmar que o indicador recebe o crédito e que a cobrança seguinte dele é estornada de verdade).

### Ranking mensal entre alunos implementado (item 15, Fase D — 2026-07-19)

### IA de interpretação de relatório implementada (plano Elite, 2026-07-19)

Fecha o item 3 da Fase A ("Diferenciação de plano + preço customizado") — até aqui o Elite só se diferenciava do Pro pelo limite de alunos ilimitado (white-label já era liberado nos dois). Pré-requisito de dado (água/sono do aluno) tinha sido resolvido mais cedo na mesma sessão de 2026-07-19 (ver seção "Coleta de dados do aluno" logo abaixo).

**Decisões fechadas com o usuário antes de codar** (as 4 perguntas em aberto do planejamento anterior): (1) gating com upsell — o botão aparece pra todo mundo, mas quem não é Elite vê um aviso de recurso exclusivo em vez do botão funcional (mesmo padrão visual de `whiteLabelLock` em `perfil.html`); (2) UI em seção adicional, lado a lado do relatório de texto determinístico — não substitui, os dois convivem; (3) resultado só aparece na hora, sem persistência em tabela nova (mesmo padrão do relatório de texto); (4) conteúdo cobre os 4 blocos que o usuário escolheu: adesão/consistência, sinais de alerta de bem-estar (água/sono), evolução física/carga, e uma sugestão prática de próximo passo.

- **Edge Function `interpret-report`** (`supabase/functions/interpret-report/index.ts`, deployada em produção): mesmo padrão de `generate-workout` (JWT do profissional escopa a leitura via RLS, sem service role key), mas sem tool-calling — um prompt, uma resposta em texto. Busca `training_protocols`/`training_history` (últimas 30 sessões), `physical_assessments` (só finalizadas), `student_anamnese` e `student_checkins` (últimos 30 dias). **Toda a aritmética é calculada no servidor antes de chamar a Claude** (adesão %, evolução de carga por exercício, evolução de peso/%gordura entre avaliações, média de água vs. meta, dias de sono ruim/curto) — a IA só recebe os números já prontos em JSON e escreve a interpretação em texto corrido, mesma divisão de responsabilidade já usada em `generate-workout` (determinístico faz conta, IA só decide/redige). **Gating checado no servidor também**, não só no front — chamar a function direto com um profissional não-Elite retorna 400 com mensagem clara, confirmado via teste (ver abaixo); é controle de custo real, não só UX.
- **`relatorios.html`**: novo card "Interpretação com IA" abaixo do relatório de texto — mostra aviso de upsell pra Starter/Pro (`professional.plan !== 'elite'`), ou botão "Interpretar com IA" pra Elite. Resultado renderiza como texto puro (`textContent`, nunca `innerHTML` — mesma prudência contra XSS já usada no resto do app) com botões Copiar/Gerar de novo.
- **Testado ponta a ponta em produção**: sandbox `doug.sickmind@gmail.com` temporariamente promovido a `plan='elite'` via SQL direto pra teste (revertido pra `starter` logo em seguida — trava de teste, não mudança real), aluno de teste com dado bem escasso (0 sessões, 0 protocolo, 0 avaliação, 1 check-in de água). A interpretação lidou bem com a escassez — disse explicitamente "não há dado suficiente" em vez de inventar, e a sugestão prática foi corretamente "ativar o acompanhamento" em vez de um ajuste de treino inventado. Zero erro no console. Testado também o bloqueio: com o plano revertido pra `starter`, o front mostra o aviso de upsell, e uma chamada direta à Edge Function (contornando o front) retorna 400 com "Interpretação por IA é exclusiva do plano Elite." — confirma que o gate real está no servidor.
- **Achado durante o teste, não um bug do código novo**: depois do primeiro deploy, o navegador continuou servindo a versão antiga de `relatorios.html` mesmo após reload — o service worker (`sw.js`, registrado nessa página desde a extensão do PWA pro painel do profissional) e o cache HTTP comum do navegador seguravam a versão pré-deploy. Resolvido nesse teste específico com `unregister()`/`caches.delete()` + `?v=` na URL pra forçar bypass. **Vale lembrar em qualquer teste futuro logo depois de um deploy**: se uma mudança em página do profissional "não aparece" mesmo com o deploy confirmado, checar isso antes de assumir bug — mesma categoria de falso-alarme já registrada outras vezes neste arquivo (aba antiga aberto antes do deploy, JS rodando em memória).
- **Não testado ainda**: qualidade da interpretação com dado rico de verdade (histórico de treino cheio, múltiplas avaliações físicas, semanas de check-in) — só o caso de dado escasso foi validado. Vale reavaliar o texto quando houver um aluno real com uso consistente.
- **Custo**: sem rate-limit específico, mesma filosofia das outras features de IA — monitorar uso real no painel da Anthropic antes de construir controle.

### Coleta de dados do aluno — água e sono (2026-07-19)

Pedido do usuário: antes de partir pra "interpretação de relatório por IA" (a feature Elite ainda não construída, próximo item lógico do roadmap), faltava terminar a coleta de dados de bem-estar do aluno. - **Schema**: `supabase_24_student_checkins.sql` — uma tabela só, `student_checkins` (não duas): água e sono são ambos "check-in diário do aluno", mesmo grão (uma linha por dia, `unique(student_id, date_key)`), mais simples de consultar junto quando o relatório de IA for construído. **Diferença deliberada do Fox**: lá o sono nunca chegou a ser persistido no Supabase (só `localStorage`, o próprio código deles comenta que a tabela remota nunca foi criada) — aqui não fazia sentido copiar essa lacuna, já que o objetivo explícito desta sessão é alimentar IA depois. RLS: aluno é dono (`for all`, igual a `training_history`), profissional só lê (é o dado que a IA vai processar, nunca edita o check-in do aluno).
- **Água**: meta = peso corporal × 35ml (mesma fórmula do Fox), com fallback de 2000ml se o aluno ainda não tem avaliação física finalizada — usa o array `assessments` que o `aluno.html` já carrega no boot (pega o mais recente, array vem ordenado ascendente). UI: strip clicável na aba Nutri (`.nutri-card.water-strip`, com barra de progresso) que abre modal com input numérico + botões de atalho (+200/+300/+500/+750ml, mesmo padrão do Fox) — aditivo, só grava ao clicar Salvar. Resumo também aparece na Home, num mini-card ao lado do sono.
- **Sono**: modal de check-in que aparece automaticamente ao abrir o app (dentro de `continuarBootAposConsentimento()`, logo após `goTab('home')`), só se ainda não respondido hoje — pergunta qualidade (🙂 Bem / 😟 Mal) e duração (😴 -6h / 🙂 6-8h / 🔥 +8h), confirmar fica desabilitado até escolher os dois, mesmo padrão exato do Fox. Emoji aqui é aceitável (mesma exceção documentada da avaliação de humor pós-treino — "conteúdo expressivo pontual"). Botão "Agora não" (que o Fox não expõe na UI, mas o código deles sugere que existe via campo `skipped`) foi adicionado deliberadamente — não travar o aluno sem saída é consistente com o princípio "nunca deve atrapalhar o fluxo" já usado no design do assistente proativo; usa `sessionStorage` pra não repetir o modal na mesma aba/dia se pulado.
- **Ícone de água**: SVG próprio (`ICONS.droplet`, estilo Feather já usado no resto do app) — nunca o emoji 💧 que o Fox usa, pra manter a regra "SVG sempre, emoji só em conteúdo expressivo pontual" (água é métrica quantitativa, não sentimento — diferente do sono).
- **Bug real achado e corrigido durante a análise pré-implementação**: `buildNutri()` tinha um `return` antecipado quando não havia orientação nutricional cadastrada, o que faria a faixa de água nunca aparecer pra um aluno sem orientação ainda configurada. Corrigido renderizando a faixa de água **antes** da checagem — funciona independente de o profissional já ter configurado a Nutri ou não.
- **Testado ponta a ponta em produção, login real como aluno** (primeira vez nesta sessão que isso foi possível — ver "Acesso self-service de teste" abaixo): consentimento LGPD → modal de sono aparece automaticamente, seleção habilita "Confirmar", grava certo, some e não repete no reload → faixa de água na Nutri (com fallback de 2000ml funcionando, sem avaliação ainda) → botões de atalho acumulam corretamente (+500+300=800) → salva e persiste na **mesma linha** do dia (upsert não sobrescreveu o sono já salvo) → meta recalculada certo ao inserir uma avaliação física de teste (80kg × 35 = 2800ml) → resumo da Home atualiza nos dois casos.
- **Lição de metodologia de teste (importante, achado durante o teste de RLS desta feature)**: `set role postgres` no `supabase db query` **ignora RLS completamente** (é superusuário) — só é um teste válido de segurança quando a checagem está dentro de uma função `SECURITY DEFINER` (que valida `auth.jwt()` no próprio corpo, independente de role). Pra testar uma **policy de RLS de tabela crua** (sem função no meio, como o caso de `student_checkins`), precisa `set role authenticated` junto com `set_config('request.jwt.claims', ...)` — só assim a policy é de fato avaliada. Um teste inicial com `role postgres` deu um falso positivo (pareceu que um profissional aleatório via o check-in de outro tenant); refeito com `role authenticated` confirmou isolamento correto. Vale lembrar disso em qualquer teste futuro de RLS que não passe por RPC.

- **Detecção de plataforma é 100% client-side** (o servidor não tem como saber o navegador) — `get_next_student_hint(p_platform)` recebe `'ios' | 'chromium' | 'none'` como parâmetro decidido no navegador, diferente do `get_next_hint()` do profissional que não precisa de nenhum parâmetro. Só existem 3 comportamentos reais: iOS Safari (nunca dispara `beforeinstallprompt`, só dá pra mostrar o guia visual), Chromium — Android/Windows/a maioria do desktop — (captura o evento nativo e vira um botão "Instalar" que chama `.prompt()`), e "nenhum" (já instalado, ou navegador sem suporte tipo Firefox — não mostra nada, sem tratamento especial).
- **Ícones do guia iOS**: nada de print real da Apple (risco de marca registrada, fica desatualizado a cada versão do iOS) nem imagem da web (licença incerta). Usa dois ícones do próprio Feather Icons (MIT, já é o sistema de ícones padrão do app) — "upload" pro passo de compartilhar, "plus-square" pro passo de adicionar — consistente com o resto do visual, sem risco de direito autoral.
- **`student-hints.js`** (novo arquivo na raiz, incluído só em `aluno.html`): mesmo padrão autocontido dos outros widgets. Como `aluno.html` não tem uma função central de "depois de trocar de tela", o arquivo **envolve `window.showScreen`** (a função real do app, definida no script principal) pra checar dica a cada navegação — mas **nunca durante `overview`/`exec`/`finish`** (fluxo de treino em andamento, incluindo o timer de descanso) — interromper um treino no meio com uma dica seria o oposto do que o usuário pediu.
- **Bug de SQL real encontrado e corrigido no teste**: `get_next_student_hint()` declarava uma coluna de retorno chamada `hint_key` (`returns table (hint_key text, ...)`) e também referenciava uma coluna `hint_key` da tabela `student_hint_events` sem alias dentro do corpo da função — Postgres trata o nome da coluna de retorno como uma variável implícita no plpgsql, e a referência sem qualificação virou ambígua (`column reference "hint_key" is ambiguous`). Corrigido com alias explícito (`student_hint_events e`, `e.hint_key`). O `get_next_hint()` do profissional não tinha esse problema porque delega a checagem "já mostrado?" pra uma função auxiliar separada (`hint_already_shown()`) em vez de fazer o `exists()` inline.
- **Testado**: como não existe conta de teste de aluno com OTP disponível (só a allowlist do `dev-generate-login` tem e-mail de profissional), a parte de navegador foi testada com um harness isolado que carrega o `student-hints.js` de verdade sem alteração nenhuma, só mockando a resposta das RPCs e forçando `navigator.userAgent`/o evento `beforeinstallprompt` — confirmado: renderização dos 2 passos com ícone no caso iOS, botão "Instalar" disparando o prompt só depois do evento "clicked" terminar de gravar no caso Chromium, e a supressão da dica ao entrar em `exec` (sem logar "dismissed", já que não foi o aluno que fechou). A camada SQL (as duas RPCs) foi testada separadamente contra o banco de verdade, simulando o JWT do aluno de teste (`aluno.teste.hint@example.com`, criado numa sessão anterior) — foi nesse teste que o bug de ambiguidade apareceu e foi corrigido.
- **Só 2 gatilhos por enquanto** (`instalar_pwa_ios`, `instalar_pwa_chromium`) — o framework já suporta adicionar mais (ex: primeiro treino executado, primeira avaliação vista) do mesmo jeito que foi feito pro profissional, mas não foram pedidos ainda nesta sessão.

Especificação vinda de um documento detalhado enviado pelo usuário (chamava o conceito de "Copiloto do Meu Protocolo"). - **Schema**: `supabase_21_assistant_hints.sql` — `training_protocols` ganha `origem` ('manual'/'ia', setada só no INSERT, nunca sobrescrita numa edição posterior); `professionals` ganha `last_seen_at`. Tabela `assistant_hint_events` é o log imutável (profissional só `insert`/`select` da própria conversa, nunca `update`/`delete`, mesmo padrão de `support_messages`). RPC `get_next_hint()` (`SECURITY DEFINER`, escopado por e-mail) avalia as 6 condições em ordem de prioridade fixa (não tem campo de prioridade genérico — não vale a complexidade extra pra só 6 itens) e retorna no máximo uma dica por chamada; `mark_hint_event(key, evento)` grava shown/dismissed/clicked. **Backfill critico**: a migration marca os 5 gatilhos "de primeira vez" como já mostrados pra todo profissional que já existia antes desta migration — sem isso, contas antigas veriam "bem-vindo, primeiro aluno!" pra coisas que já aconteceram há semanas.
- **Os 6 gatilhos**: primeiro acesso (nunca mostrado antes, dispara uma vez só), primeiro aluno cadastrado (≥1 aluno), primeiro treino criado (≥1 protocolo), primeira geração por IA (≥1 protocolo com `origem='ia'`), retorno após inatividade (gap desde `last_seen_at` > 7 dias — **auto-resetável**, `get_next_hint()` já atualiza `last_seen_at = now()` como efeito colateral da própria leitura, então não repete no próximo carregamento), sugestão de experimentar IA (≥3 protocolos manuais e zero de IA).
- **`assistant-hints.js`** (novo arquivo na raiz, mesmo padrão autocontido do `support-widget.js`): card flutuante no canto inferior **esquerdo** (o widget de suporte já ocupa o direito). Expõe `window.__checkAssistantHint` pra páginas chamarem manualmente logo após uma ação relevante (feedback imediato, sem esperar o próximo carregamento) — usado em `alunos.html` (após cadastrar aluno) e `treinos.html` (após salvar protocolo, dentro de `saveProtocol()`).
- **`treinos.html`**: variável `lastCreationSource` (não confundir com `creationMode`, que só indica qual painel está aberto) — setada `'ia'` só depois de `doGenerateAi()` ter sucesso de verdade, `'manual'` ao abrir criação manual ou usar "Aplicar modelo"/"Duplicar" (esses dois ficam sempre visíveis, não são exclusivos do modo manual, e o conteúdo deles não vem da IA mesmo que o wizard estivesse aberto). `origem` só entra no payload do **insert** em `saveProtocol()` — nunca no `update`, pra não sobrescrever a origem original numa edição posterior.
- **Dois bugs reais encontrados e corrigidos durante o teste ao vivo** (ambos em `assistant-hints.js`, nenhum no schema):
  1. `supa.rpc(...)` chamado sem `await`/`.then()` nunca dispara a requisição de verdade no `supabase-js` (é um "thenable" preguiçoso — só executa quando algo consome a Promise). Os três registros de evento (`shown`/`dismissed`/`clicked`) eram fire-and-forget sem isso — nenhum estava sendo gravado. Corrigido com um helper `logHintEvent()` que sempre encadeia `.then()`.
  2. Depois do primeiro fix, o clique em botão com navegação ainda não gravava `clicked`: `hide()` zera `currentKey` e estava sendo chamado **antes** do valor ser lido pra registrar o evento — e navegar embora (`window.location.href`) cancela a requisição em voo mesmo com o `.then()` certo. Corrigido capturando a chave numa variável local antes de `hide()`, e esperando (`await`) o registro terminar antes de navegar.
- **Testado ponta a ponta em produção** (mesma conta sandbox `doug.sickmind@gmail.com`, resetada pra simular conta nova): os 3 gatilhos que dependem de ação do usuário (`primeiro_acesso`, `primeiro_aluno`, `primeiro_treino`) testados via navegador de verdade — card aparece, clique registra `clicked` e navega, `origem='manual'` gravado certo no protocolo criado manualmente. Os outros 3 (`primeira_ia`, `sugerir_ia`, `retorno_inatividade`) testados via simulação de JWT direto no banco (mesma técnica usada pra validar as RPCs do master) — mais barato que gerar treino por IA de verdade só pra testar uma condição de contagem. Confirmado também que `retorno_inatividade` não repete na chamada seguinte (auto-reset funcionando).
- **Não testado**: a interação entre dois profissionais reais vendo dicas diferentes ao mesmo tempo (baixo risco, cada linha de `assistant_hint_events`/`get_next_hint()` já é escopada por `professional_id` desde o desenho); e o comportamento visual em telas sem `.bottom-nav` (`treinos.html`/`avaliacoes.html`/`nutri.html`) não foi conferido com uma captura de tela de verdade, só via inspeção de DOM (mesma limitação de screenshot já registrada na sessão anterior).

### Fase B fechada: Cloudflare, rate limit, UptimeRobot e Sentry (2026-07-18)

Últimos itens em aberto da Fase B (ver checklist "Roadmap priorizado" no fim deste arquivo). Feito com ajuda do usuário via Claude in Chrome — dois navegadores conectados em paralelo, um logado no Supabase (`contatofoxperformance@gmail.com`) e outro no Cloudflare (`meuprotocolo1@gmail.com`); troca entre eles via `select_browser`/`list_connected_browsers`.

- **Cloudflare — proxy + headers de segurança**: o domínio `meuprotocolo.app` estava com DNS só delegado pra Cloudflare (nuvem cinza, "DNS only") — confirmado via headers HTTP (`Server: GitHub.com`, sem `cf-ray`) antes de mexer. Ativado o proxy (nuvem laranja) nos dois registros CNAME que servem o site (`@` e `www`) — os registros de e-mail do Resend (MX + 2 TXT) foram deixados como `DNS only` de propósito, proxy não se aplica a eles. Depois, criada uma Response Header Transform Rule (Rules → Transform Rules) aplicando `Strict-Transport-Security: max-age=31536000; includeSubDomains`, `X-Frame-Options: DENY` e `X-Content-Type-Options: nosniff` em toda resposta — confirmados ao vivo via `curl -I`.
- **Efeito colateral do proxy, investigado e descartado**: logo depois de ligar o proxy, um teste flagrou a Cloudflare injetando sozinha um script de analytics (`static.cloudflareinsights.com/beacon.min.js`) nas páginas, bloqueado pelo CSP. Investigação em Analytics → Web Analytics mostrou que o "Real User Measurements (RUM)" da zona está **desligado** (botão era "Enable", não "Disable") — e checagens repetidas via curl confirmaram que o script já não aparece mais. Foi um artefato temporário de zona recém-proxiada, resolvido sozinho; nada foi alterado no painel por causa disso.
- **Rate limiting do Supabase Auth**: conferido em Authentication → Rate Limits — todos os valores são o padrão do Supabase (30 e-mails/hora **por projeto inteiro, não por usuário** — pode virar gargalo de UX com escala, não é problema de segurança; 30 verificações de OTP/5min por IP; etc.). Adequado pro estágio atual, nada foi alterado.
- **UptimeRobot**: monitor HTTP(S) criado pra `https://meuprotocolo.app`, alerta por e-mail confirmado via "Test Notification".
- **Sentry**: projeto criado (plataforma Browser JavaScript, "Nope, Vanilla" — sem framework), método de instalação "Loader Script" (mais simples que DSN manual, já vem com o projeto embutido). Script `https://js.sentry-cdn.com/dd3ca3ee7a720284b2a9bf8240b29158.min.js` adicionado no `<head>` das 13 páginas do repo, carregando antes do SDK do Supabase pra capturar erro desde o boot. CSP de todas as páginas ganhou `js.sentry-cdn.com` em `script-src` e o host de ingest específico do projeto (`o4511757250199552.ingest.us.sentry.io`) em `connect-src` — testado ao vivo, `window.Sentry` confirmado presente e nenhum erro de CSP no console.
- **Bug encontrado durante o teste de conquistas do aluno (não relacionado ao Sentry, achado em paralelo)**: o toast de "Conquista desbloqueada" mostrava o código SVG do ícone como texto puro em vez de desenhar o ícone — `toast()` sempre usava `textContent` (correto pra mensagens de erro/status, que são texto puro), mas esse call site específico monta `${b.ico} Conquista: ...` esperando que o SVG (trocado de emoji pra SVG numa sessão de UI polish anterior) fosse renderizado como HTML. Corrigido com um segundo parâmetro opcional `toast(msg, isHtml)`, usado só nesse call site; nome da conquista continua escapado por segurança.
- **Pendências que ficam de fora deliberadamente**: revisão jurídica de termos/privacidade (não-técnico, precisa do usuário + cliente 0) e backup/PITR do Supabase (pendência de fim de projeto, ver seção própria mais abaixo).

### Cardio como tipo de exercício à parte (2026-07-18)

### Webhook + assinatura Mercado Pago (2026-07-13) — Fase A, item 1

Decisão de arquitetura: **Preapproval (assinatura nativa)**, não cobrança avulsa repetida via cron. Motivo: o profissional sempre paga em cartão (sem boleto/Pix), então o Mercado Pago pode cuidar sozinho do agendamento mensal e do retry em cartão recusado — evita ter que construir cron + lógica de dunning própria antes da Fase B (monitoramento) existir. O objeto `preapproval` aceita um bloco `free_trial` nativo (frequência + duração), que mapeia direto pra "trial de 14 dias com cartão cadastrado desde o cadastro" do master doc seção 4: o cartão é autorizado no onboarding, o Mercado Pago só cobra depois do 14º dia — nosso lado só escuta o webhook e sincroniza status, nunca dispara cobrança.

- **Schema**: `supabase_14_billing.sql` — `professionals` ganha `mp_preapproval_id`, `mp_subscription_status`, `ultima_cobranca_status`, `ultima_cobranca_em`; nova tabela `billing_events` (log/idempotência dos webhooks recebidos — o Mercado Pago pode reenviar a mesma notificação mais de uma vez, `unique (mp_id, mp_type)` evita reprocessar). **Ainda não aplicada em produção** — precisa rodar via SQL Editor do Supabase ou `supabase db query --linked --file` localmente.
- **Edge Function `mercadopago-webhook`**: recebe as notificações, valida o header `x-signature` (HMAC-SHA256 sobre o manifest `id:{data.id};request-id:{x-request-id};ts:{ts};`, conforme documentado pelo Mercado Pago — nunca confia no corpo da notificação sozinho), e por segurança **busca o recurso de novo na API deles** (`GET /preapproval/{id}` ou `GET /v1/payments/{id}`) antes de atualizar `professionals`. Usa `external_reference` (setado como o `professionals.id` na hora de criar a assinatura) pra saber qual profissional atualizar. **Precisa de deploy com `--no-verify-jwt`** (o Mercado Pago não manda JWT do Supabase, só o `x-signature` próprio dele) — sem essa flag toda notificação seria rejeitada com 401 antes mesmo de chegar no código.
- **Edge Function `mercadopago-create-preapproval`**: chamada pelo onboarding depois que o cartão é tokenizado no navegador (nunca recebe número de cartão cru); cria a assinatura via `POST /preapproval` com `auto_recurring.free_trial` de 14 dias e valor fixo por plano (`PLAN_PRICES` — `starter`/`pro`/`elite` batendo com a tabela de preços do topo deste arquivo). **Nota**: quando o item 3 (diferenciação de plano + `valor_customizado`) for implementado, essa função precisa passar a ler o valor de lá em vez do mapa fixo.
- **`onboarding.html`**: virou fluxo de 2 passos — nome (como já era) → cartão (Mercado Pago "Secure Fields" via `mp.fields.create`/`createCardToken`, SDK `sdk.mercadopago.com/js/v2`, CPF do titular obrigatório). Se a conta já existe sem `mp_preapproval_id` (ex: fechou o navegador no meio), pula direto pro passo do cartão em vez de pedir o nome de novo.

Conclusão prática: **qualquer comando `supabase` que fale com a API remota (`link`, `functions deploy`, `secrets set`, `db push` contra o projeto remoto) só funciona rodando localmente** (notebook, onde a CLI já funciona bem pro projeto Fox) ou numa sessão de Claude Code que rode nesse ambiente local. SQL direto (colar no SQL Editor do navegador) e o próprio deploy do site (GitHub Pages) não têm esse problema — continuam funcionando normal de qualquer sessão.
